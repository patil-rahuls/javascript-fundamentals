## `let`, `const` & `var` - 4 Differences

### 1. Re-assignment
> `let` and `var` variables can be re-assigned.
>
> `const` variables cannot be re-assigned.

```javascript
let a = 10;
a = 20;

var b = "rahul";
b = "I just got re-assigned";
```

Variables declared using `const` cannot be re-assigned.

```javascript
const c = 4.5;
c = false;    // ŌØī Incorrect
// 'Uncaught TypeError: Assignment to constant variable'
```

This makes `const` the best choice for declaring variables that donŌĆÖt need to change. It prevents mistakes such as accidental re-assignment.

However, for Objects `const` does allow mutation (NOT re-assignment).

```javascript
// Ō£ģ Mutation is allowed.
const myArr = [];
myArr.push(1);      //
myArr[9] = 10;      // Mutation is allowed


// ŌØī Re-assignment is not allowed for Objects declared with `const`.
myArr = [1, 2, 3];
// 'Uncaught TypeError: Assignment to constant variable'
```

### 2. Scope
> `let` and `const` variables are ***`BLOCK`*** scoped.
>
> `var` variables are ***`FUNCTION`*** scoped.

Scope refers to the visibility or accessibility of variables, functions, and objects within different parts of a program.

It determines where in the program a particular variable or a function can be accessed and manipulated.

`let` and `const` variables are scoped to the immediate enclosing block denoted by ```{ }``` (hence the ***`BLOCK`*** scope).

`var` variables are scoped to the immediate function body (hence the ***`FUNCTION`***  scope).

In general, a block is any code within curly braces ```{ }```, including functions, loops and conditional statements.

> Example 1

```javascript
function run() {
  // Function scope
  var foo = "Foo";
  let bar = "Bar";

  console.log(foo, bar);
  // 'Foo Bar'

  {
    // Block scope
    var moo = "Mooo";
    let baz = "Bazz";

    console.log(moo, baz);
    // 'Mooo Bazz'
  }

  console.log(moo);
  // 'Mooo'

  console.log(baz);
  // 'Uncaught ReferenceError: baz is not defined'
}

run();
```

&nbsp;
> Example 2

```javascript
var x = "global scope";

function foo() {
  var x = "function scope";
  console.log(x);
}

{
  let x = "block scope";
  // variable x above is only accessible in this enclosing block
}

foo();
// 'function scope'

console.log(x);
// 'global scope'
```

&nbsp;
> Example 3: Scope changes when a variable is declared without `let`, `var` or `const` keyword.

```javascript
{
  let a = 9;
  b = 8;
  // b is not declared using any of the 'let', 'const', 'var' keywords and by default becomes globally scoped.
}

console.log(a);
// 'ReferenceError: a is not defined'
// (Because, let is block scoped)

console.log(b);
// '8'
// (Because, b is globally scoped)
```

'Globally scoped' means the variable is accessible by the global object.

In Browser, we can access the variable `b` using the global object - `window`.

```javascript
console.log(window.b);
```

In Node JS runtime, we can access 'b' using `global` object. _(The name itself is 'global')_

```javascript
console.log(global.b);
```

### 3. Hoisting
> `var` variables are hoisted to the top of their respective scope with the value `undefined` before the code is run.
>
> `let` & `const` variables are put in `Temporal Dead Zone` until their initialization is processed.

**Hoisting** refers to the behavior of variable and function declarations being moved to the top of their scope.

`var` variables are always hoisted to the top of their respective scope with the value `undefined` before the code is run.

They are accessible in their enclosing function scope even before they are declared.

> Example 1: `var` variables are hoisted.

```javascript
function foo() {
  console.log(x);
  // undefined

  var x = "Rahul";

  console.log(x);
  // 'Rahul'
}

foo();
```
`let` & `const` variables are also hoisted but they are said to be in`ŌĆśTemporal Dead Zone` from the start of the block until their initialization is processed.

&nbsp;
> Example 2: `let` variables are not hoisted.

```javascript
function checkHoisting() {
  console.log(x);
  // 'Uncaught ReferenceError: Cannot access 'x' before initialization'

  let x = "Rahul";

  console.log(x);
  console.log(y);
  // 'Uncaught ReferenceError: 'y' is not defined'
}

checkHoisting();
```
&nbsp;
> WHY TDZ exist ?
>
> Ans: _to identify the type of issue/bug_

Observe the error message in the Example 2:

`"Cannot access 'x' before initialization`

It is because the code is certain that variable `x` is initialized later in the code.

Hence it is kept in `TDZ`. Thus we can identify the issue from error message created by TDZ and it makes it easier to avoid errors/bugs which are hard to find.

In case of `var`, we would get:

`'y' is not defined`

which seems to be an identical error in most of other cases, and it becomes difficult to find the actual bug.

Also, another reason why TDZ exists, is to make `const` variables work as intended. i.e. to throw error if `const`variables are not initialized while declaring.

&nbsp;
> Why `var` varibles can be more prone to errors.

All the following examples work without giving any error, but they look bad and erroneous.

```javascript
var name = "Rahul";
var name = "Raj";
// This works without giving any error.
```

`var` variables are initialized with the value 'undefined' and moved to the top of their scope (hoisting).

Functionally, the code shown above executes like this:

```javascript
var name = undefined;
name = "Rahul";
name = "Raj";
```

&nbsp;

Another example:

```javascript
b = 25;
var b;
console.log(b);
// 25

// Functionally, the code executes like this:
var b = undefined;
b = 25;
console.log(b);
```

&nbsp;

Another example of bad-looking yet working code:

```javascript
var a = 1;
var a = 2;
console.log(a);
// 2

var a;
console.log(a);
// 2

// Functionally, the code executes like this:
var a = undefined;
a = 1;
a = 2;
console.log(a);

var a; // This does not change the value of 'a' because it is already initialized.
console.log(a);
```

&nbsp;
> IMP - Function declarations are hoisted before any `var` initializer gets evaluated.

```javascript
var a = 1;
function a() {

}

console.log(a);
// 1

//Functionally, the code executes like this:
function a() {} // fn. declarations hoisted.
var a = 1;      // then 'a' is initialized
console.log(a); // '1'
```

&nbsp;

Another interesting example.

```javascript
try {
  f = 100;
  throw 1;
} catch (e) {
  var f = 2;
}

console.log(f);
// 100
```

The `var` variable `f` is hoisted to outside the `catch` block with value `undefined`, but any value assigned within the `catch` block is not visible outside.

Hence it returns the last assigned value which is 100. WOW !

### 4. Creating Global object property
> At the top level, `var` variables create a property on the global object.
>
>`let` and `const` variables do not.

```javascript
var foo = "Foo";    // globally scoped
let bar = "Bar";    // not globally scoped
baz = "Baz";        // globally scoped

// In browser:
console.log(window.foo);
// 'Foo'
console.log(window.baz);
// 'Baz'
console.log(window.bar);
// undefined

// In Node JS runtime:
console.log(global.foo);
// 'Foo'
console.log(global.baz);
// 'Baz'
console.log(global.bar);
// undefined
```

Hence, variables declared using `let` and `const` are not accessible to global objects like the `window` or `global` object.

The purpose of `let` and `const` statements is only to free up memory when not needed in a certain block.
